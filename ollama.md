# Ollama - локальные LLM

[Ollama](https://ollama.com/) -  это инструмент для простого локального запуска больших языковых моделей (LLM) одной командой. Его часто сравнивают с Docker — но применительно к AI‑моделям: достаточно выполнить одну команду, и модель запускается без необходимости разбираться в зависимостях, конфигурациях и драйверах.

Пример команды (папка в Windows - `%USERPROFILE%\AppData\Local\Programs\Ollama`):

```shell
ollama run llama2
```

Ollama работает с рядом известных открытых LLM, в том числе:

- LLaMA (включая Llama 3)
- Mistral
- CodeLlama

Ключевое свойство - автоматическая оптимизация:

- определяет доступные ресурсы (CPU/GPU)
- подстраивает загрузку модели под возможности оборудования
- поддерживает квантование и оптимизацию моделей

Небольшие модели можно запускать и на бюджетных процессорах, например, на Core i3 13100, но рекомендуется использовать только 3B‑модели (например, llama3.2:3b, phi3:mini):

```shell
# Запуск Phi‑3 Mini
ollama run phi3:mini

# Запуск Llama 3.2 3B
ollama run llama3.2:3b

# Запуск квантованной Mistral 7B
ollama run mistral:7b-instruct-v0.1-q4_k_m

ollama run gemma4:e2b
```

Core i3‑13100 подходит для экспериментов с Ollama и LLM, но не более того.

Настройки для повышения производительности:

- количество параллельных запросов не более 1–2
- квантованные версии моделей (формат GGUF, 4–8 бит)
- все фоновые ресурсоёмкие приложения должны быть закрыты
- для снижения нагрузки следует уменьшить размер контекста. Например: `--num-ctx 2048`

## Альтернативы

Одной из популярных альтернатив Ollama является [LM Studio](https://lmstudio.ai/).

Доступ клиент для Windows. Ключевое отличие состоит в том, что если Ollama - это минимализм, то LM Studio больше похож на Visual Studio Code - у него больше настроек, лучше поиск и вообще больше информации о процессах, которые связаны с использованием LLM.

В качестве альтернативы также можно использовать [Lemonade](https://lemonade-server.ai/).

[llama.cpp](https://github.com/ggml-org/llama.cpp) считается одной из наиболее производительных приложений для запуска inference LLM. Поддерживает не только GPU Nvidia, но и GPU AMD через HIP, а также Moore Threads GPU через MUSA.

### Тарифные планы

Проект Ollama предлагает несколько планов для использования моделей, включая: Free, Pro и Max. Все три тарифа позволяют использовать облачные модели, но Free план самый медленный и на нём больше всего ограничений. На максимальном плане можно использовать несколько машин для выполнения параллельной работы.

Наличие тарифных планов означает, что для полноценного использования Ollama необходимо выполнить аутентификацию пользователя (может быть использован аккаунт GitHub).

Хостинг моделей допустим только на тарифных планах Pro и Max.

## Hugging Face

[Hugging Face](https://huggingface.co/) - это тоже самое для машинного обучения, что GitHub для кода. На Hugging Face размещаются ИИ модели, решающие различные задачи. В частности, есть много узкоспециализированных ИИ моделей, например, только для генерации кода на ИИ. Узкоспециализированные модели могут эффективно решать сложные задачи, расходуя кратно меньшее количество ресурсов, чем при использовании универсальных моделей.

>Hugging Face после нескольких лет плодотворного сотрудничества купила команду llama.cpp. Предполагается, что ПО останется открытым, но у разработчиков появится больше ресурсов для совершенствования продуктов.

В репозитариях Hugging Face находятся как модели, так и Datasets.

>Hugging Face - это эмодзи "обнимашка". Миссия Hugging Face: "ИИ для исследователей, демократизация ИИ".

Для запуска моделей с Hugging Face необходима Ollama. Можно зарузить Docker-контейнер, но для этого нужне API-ключ (DASHSCOPE_API_KEY), а также поддержку containerd engine в Docker.

Благодаря использованию карточек моделей и поиска, на Hugging Face довольно удобно искать специализированные модели.

## Установка Ollama

Команда установки Ollama в Linux:

```shell
curl -fsSL https://ollama.com/install.sh | sh
```

Проверка версии:

```shell
ollama --version
```

### Установка в Microsoft Windows

Инсталлятор клиента Ollama в Microsoft Windows загружается напрямую с сайта. Его размер составляет 1.2 ГБ. Настроек установки на конкретный носитель нет, т.е. клиент устанавливается безальтернативно на диск c. Доступно всего несколько моделей. Размер модели gemma3:4b - 3.1 ГБ. Это сбалансированная модель от подразделения Google DeepMind (по сути, это упрощённая модель Gemini). Линейка Gemma призвана обеспечить высокую производительность при относительно небольших размерах моделей, что позволяет запускать их на локальном оборудовании — смартфонах, ноутбуках, рабочих станциях с GPU или TPU.

Ollama выполняет модель gemma3:4b вполне шустро, со скоростью чуть медленнее вывода на Duck.io в web, на компьютере с процессором Core i3-13100, без дискретной графики. При этом процессор нагревается до 60+ градусов.

Приложения "ollama app.exe" и "ollama.exe" находятся в папке: `%USERPROFILE%\AppData\Local\Programs\Ollama`

На портале Hugging Face можно найти модель в формате GGUF. На странице модели нужно выбрать "Use this model" и "Ollama", а затем получить команду для загрузки в консоли Ollama. Например:

```shell
ollama run hf.co/mistralai/Ministral-3-8B-Instruct-2512-GGUF:Q4_K_M
```

В описании модели есть список аппаратной совместимости. LLM любят VRAM и FP8: "_The Ministral 3 family is designed for edge deployment, capable of running on a wide range of hardware. Ministral 3 8B can even be deployed locally, capable of fitting in 12GB of VRAM in FP8, and less if further quantized._"

Для задач edge deployment минимальная аппаратная рекомендация - использование дискретной видео-карты с 16 ГБ ОЗУ VRAM.

Эта крошечная модель работает на Core i3 13100 очень быстро. Существенное ограничение - поддержка только английского языка. Для теста попросил проанализировать LLM код моего короткого приложения на Python:

```py
import numpy as np
import pandas as pd
import ast
import matplotlib.pyplot as plt
import seaborn as sns

df = pd.read_csv('dota_hero_stats.csv')

def parse_roles(x):
    try:
        parsed = ast.literal_eval(x)
        if isinstance(parsed, (list, tuple, set)):
            return list(parsed)
    except Exception:
        return []
    return []

lengths = df['roles'].apply(lambda x: len(parse_roles(x)))

sns.set(style="whitegrid")

plt.figure(figsize=(8, 4))
sns.histplot(lengths, bins=range(0, max(lengths)+2), discrete=True)
plt.xlabel("Роли")
plt.ylabel("Количество")
plt.title("Количество ролей пользователей")
plt.xticks(range(0, max(lengths)+1))
plt.tight_layout()
plt.show()
```

Ответы LLM:

_The code above is a Python script that loads and manipulates data from two different CSV files. The first file is named 'dota_hero_stats.csv' which contains Dota's player roles, while the second file is called 'roles.csv'._

Галлюцинация: код считывает только один файл - "dota_hero_stats.csv". Код работает с колонкой "Roles" из CSV-файла, а не со вторым файлом.

_Firstly, it uses the numpy module to load the data from the 'dota_hero_stats.csv' file, which contains Dota's player roles_

Галлюцинация: загружает данные из CSV-файла не numpy, а Pandas.

_After loading the data, it uses the pandas module to manipulate this data as it is and creates a series from it. The result is a column with the number of roles for each user._

Да, это верная оценка.

_Next, it applies a list comprehension to iterate over the 'roles' columns in the 'dota_hero_stats.csv' file and stores them into the 'lengths' variable._

Галлюцинация: ошибка, используется не list comprehension, а лямбда.

_Then, it uses seaborn's seaborn library to generate a white grid histrogram plot to show how many users have each role._

Да, корректно.

В целом, эта крошечная модель сделала много существенных, скорее даже критичных ошибок.


## Qwen 3.5 для маломощных ПК

Модель вышла в марте 2026 года. Установка через терминал Ollama:

```shell
ollama run qwen3.5:9b
```

Размер модели 6,6 ГБ. Модель относится к Small LLM для Edge Deployment (т.е. размещение на ноутбуках и производительных смартфонах), располагаясь максимально близко к границе с Middle-моделями.

>-Tiny (крошечные модели) - 0,5-2 миллиард параметров. Реализуют базовые возможности общения
>-Small (малые модели) - от 2 до 13 миллиардов параметров. Ориентированы на edge deployment
>-Middle (средние модели) - от 13 до 70 миллиардов параметров. Часто работают локально на серверах и мощных ПК с производительной дискретной графикой
>-Big (большие модели) — от ~70 миллиардов параметров и выше. Максимальное качество, но требуют огромных вычислительных ресурсов
>
>Чёткой общепринятой границы не существует — разные исследователи и компании используют свои пороги. Количество параметров не всегда напрямую определяет качество: архитектура, данные и обучение играют не меньшую роль. Отдельно стоит упомянуть MoE-модели (Mixture of Experts) — например, Mixtral 8x7B имеет 47B параметров суммарно, но активирует только ~13B за раз.

Это думающая модель, подготовленная для медленных компьютеров и смарт-фонов. На моей машине Core i3-13100, генерация составляет 2-3 слова в секунду. Вентилятор работает на 82%, температура ~62 градуса Цельсия.

Результат работы отличный - Qwen очень подробно разъясняет, что делает мой код и при этом разъяснения корректные.

Единственное существенное ограничение - скорость работы. На видео-карте с VRAM, производительность была бы, скорее всего, комфортной для практической работы.

Важный нюанс состоит в том, что существует возможность подключить LLM к IDE и работать в режиме _vibe-coding_.

## VibeThinker

VibeThinker — это семейство компактных языковых моделей (LLM), которые прославились тем, что показывают неожиданно высокую эффективность в логике, математике и программировании при очень небольшом числе параметров.

Основные версии и их особенности:

- VibeThinker‑1.5B (1,5 млрд параметров) — стала известной как «микро‑модель уровня GPT‑OSS‑20B»: при скромном размере она обгоняет значительно более крупные модели в ряде тестов. Например, в математических бенчмарках AIME24, AIME25 и HMMT25 её результаты выше, чем у DeepSeek R1 (у которой параметров в 400 раз больше). В задачах генерации кода (LiveCodeBench v5/v6) модель тоже демонстрирует сильные показатели
- VibeThinker‑3B (3 млрд параметров) — развивает ту же идею: делает упор на эффективное «сжатие» логического мышления, а не на накопление энциклопедических знаний. В практических тестах (в том числе на LeetCode) она способна решать сложные алгоритмические задачи и конкурировать с гораздо более тяжёлыми моделями

Особенности модели:

- Высокая эффективность на задачу при малом размере. VibeThinker доказывает, что для задач с проверяемым ответом (математика, код, логические рассуждения) не обязательно нужны гигантские нейросети
- Особый метод обучения. В основе лежит подход "спектр‑в‑сигнал" (SSP): на этапе обучения с учителем модель осваивает множество разных способов решения, а затем с помощью обучения с подкреплением отбирает самые надёжные стратегии
- Специализация на STEM и коде. Лучше всего модель проявляет себя в математике, алгоритмах и программировании

Установить модель можно используя CLI:

```shell
ollama run hf.co/KakTakOne/VibeThinker-3B-GGUF:Q4_K_M
```

При запуске на GPU RTX 5060 занимает 4 ГБ VRAM.

### nvidia‑smi

NVIDIA System Management Interface — утилита командной строки для мониторинга и управления графическими процессорами (GPU) NVIDIA. Она поставляется вместе с драйверами NVIDIA.

Ключевые метрики мониторинга:

- GPU‑Util — доля времени (за последние 1–2 секунды), в течение которого хотя бы один CUDA‑kernel выполнялся на GPU (не равноценно "проценту загрузки вычислительных ядер")
- Memory‑Usage — использование видеопамяти (VRAM): сколько занято и каков общий объём
- Temp — текущая температура GPU
- Power Draw — текущее энергопотребление (в ваттах), важный индикатор реальной нагрузки

## Что даёт использование дискретной графики в Ollama

- Скорость генерации токенов — главное преимущество. GPU обладает тысячами параллельных вычислительных ядер, что идеально подходит для матричных операций, которые лежат в основе работы трансформерных моделей. Скорость генерации может вырасти в 5–20 раз
- Загрузка модели в VRAM — если модель целиком помещается в видеопамять, все вычисления выполняются без обращения к оперативной памяти или диску. Это критически важно для низкой задержки (latency)
- Поддержка более крупных моделей — видеокарты с большим объёмом VRAM (например, 16–24 ГБ) позволяют запускать модели на 13B, 30B параметров и выше без квантизации или с минимальной потерей качества
- Частичная выгрузка на GPU (GPU offloading) — если модель не помещается целиком в VRAM, Ollama позволяет выгрузить в GPU только часть слоёв (параметр num_gpu). Даже частичная выгрузка заметно ускоряет работу по сравнению с чистым CPU-режимом

Отличие VRAM от DRAM:

- Физическая архитектура и шина: VRAM современных GPU использует стандарт GDDR6/GDDR6X или HBM2/HBM3. Ширина шины памяти у видеокарт составляет 128–384 бита и выше, тогда как у системной DDR5 — всего 64 бита на канал. Это позволяет GPU перекачивать огромные объёмы данных за один такт
- Пропускная способность: Это ключевое отличие. Современная GDDR6X обеспечивает 900–1000 ГБ/с, а HBM3 — свыше 3 ТБ/с. Системная DDR5 даже в двухканальном режиме даёт около 80–100 ГБ/с. Разница — в десятки раз
- Латентность: Системная DRAM оптимизирована под низкую латентность при последовательном доступе к случайным адресам — это важно для CPU с его ветвлениями и непредсказуемым порядком обращений к памяти. VRAM же жертвует латентностью в пользу пропускной способности, потому что GPU работает с тысячами потоков параллельно и скрывает задержки за счёт переключения между ними
- Физическое расположение: VRAM распаяна прямо на плате видеокарты максимально близко к GPU-чипу, что физически сокращает путь сигнала. HBM и вовсе монтируется в одном корпусе с чипом через кремниевый интерпозер (2.5D-упаковка)
- Назначение и доступность: Системная DRAM — универсальная, доступна всем процессам ОС. VRAM — выделенный ресурс GPU, недоступный напрямую для CPU без явной операции копирования через шину PCIe (что само по себе является узким местом)
- Стоимость: VRAM значительно дороже в пересчёте на гигабайт именно из-за высокочастотной архитектуры и специализированного производства.
Если коротко: DRAM — это универсальный и доступный склад, а VRAM — узкоспециализированный высокоскоростной конвейер, заточенный под параллельную обработку матричных вычислений, что и делает её незаменимой для нейросетей и графики

Для сравнения несколько вариантов видео-карт, совместимых с Ollama, доступных в Москве в марте 2026 г.:

- GeForce RTX 5060: NVIDIA Blackwell, 8 ГБ памяти GDDR7 448 Гбайт/сек, эффективная частота памяти 28 ГГц (33 тыс. рублей)
- GeForce RTX 5060 Ti: NVIDIA Blackwell, 8 ГБ памяти GDDR7 448 Гбайт/сек, эффективная частота памяти 28 ГГц (50 тыс. рублей)
- Radeon RX 7600 XT: AMD RDNA 3, 16 ГБ памяти GDDR6 288 Гбайт/сек, эффективная частота памяти 18 ГГц (36 тыс. рублей)
- Radeon RX 9060 XT: AMD RDNA 4, 16 ГБ памяти GDDR6 320 Гбайт/сек, эффективная частота памяти 20 ГГц (42 тыс. рублей)

Практические ориентиры по VRAM:

— 4–6 ГБ: модели 7B в формате Q4
— 8–12 ГБ: модели 13B в формате Q4–Q5

Мнения из интернета: в типичных ИИ-нагрузках CUDA часть превосходит альтернативные платформы (например, ROCm от AMD) на 10-30%. Это связано с многолетней оптимизацией под конкретные архитектуры GPU, поддержкой специализированных форматов данных (FP16, BF16, FP8) и тензорных ядер.

## Квантизация/Квантование

Квантизация (quantization) в контексте нейросетей и ML — это процесс снижения точности числового представления весов и активаций модели с целью уменьшения её размера и ускорения инференса.

Обычно веса нейросети хранятся в формате float32 — это 32 бита на каждое число. Квантизация переводит их в менее точный формат: float16, bfloat16, int8, int4 или даже int2/int1. Например, если модель весит 14 ГБ в float32, то в int4 она займёт около 3.5 ГБ.

Главная причина — вписать модель в ограниченный объём VRAM. Большие языковые модели (LLM) вроде LLaMA 70B в полной точности требуют сотни гигабайт памяти, что недоступно для потребительских видеокарт. Квантизация позволяет запустить такую модель на обычном железе. Кроме того, уменьшается объём данных, перекачиваемых по шине памяти, а значит растёт скорость инференса — особенно там, где узким местом является пропускная способность VRAM.

Чем агрессивнее квантизация, тем сильнее "размываются" веса — модель начинает хуже рассуждать, теряет тонкие различия в знаниях, могут появляться галлюцинации и деградация в сложных задачах. int8 — почти незаметно. int4 — заметно на сложных задачах. int2 — модель деградирует существенно.

Форматы квантизации GGUF (llama.cpp), GPTQ и AWQ — наиболее распространённые. Они отличаются методом компрессии и способом калибровки, но цель одна: минимизировать потерю качества при максимальном сжатии.

Если коротко: квантизованная модель — это компромисс между качеством и доступностью. Она менее требовательна к железу, но чуть менее умна, чем оригинал в полной точности.

## Подсчёт объёма VRAM необходимого для запуска модели

Для оценки применимости модели часто используют формулу:

```
Требуемая VRAM = Веса модели (квантованные) + KV‑кэш + Системные накладные расходы
```

Типичные значения для 8B‑модели с 4‑битной квантизацией:

- Веса модели: ~4 ГБ (0,5 байта на параметр)
- KV‑кэш (для контекста 8K): 2–3 ГБ
- Системные накладные расходы (буферы, обработка): 1–2 ГБ.

Итого: 7–9 млрд. параметров — укладывается в лимит 8 ГБ с небольшим запасом.

Для программирования рекомендуется попробовать следующие модели:

- deepseek-coder-v2:6.7b (Q4) — специализируется на коде, ~3,8 ГБ, требует ~6 ГБ VRAM
- nemotron-nano:9b — лучший по LiveCodeBench, хорош для генерации кода. Это разработка Nvidia, см. [страницу модели на Hugging Face](https://huggingface.co/lmstudio-community/NVIDIA-Nemotron-3-Nano-30B-A3B-GGUF)

Резюме: На видеокарте с 8 ГБ VRAM следует запускать 7–8B модели с 4‑битной квантизацией (Q4_K_M). 

Длина контекста: 4K (безопасный вариант), 8K (проверять по нагрузке).

Что не запускать:

- Модели 13B+ в FP16/Q8 без агрессивного квантования и обрезки контекста
- Мультимодальные модели с высоким разрешением изображений

## Архитектура LLM

Ключевые термины:

- микшер экспертов (MoE) - архитектура нейронных сетей, в которой модель разделена на несколько специализированных подмодулей («экспертов»). Для каждого входного запроса активируется лишь часть из них — те, что лучше подходят для решения текущей задачи. Это позволяет масштабировать модель до огромного числа параметров без пропорционального роста вычислительных затрат
- KV‑кэш (Key‑Value Cache) — это механизм оптимизации инференса больших языковых моделей (LLM), который позволяет существенно ускорить генерацию текста за счёт повторного использования ранее вычисленных промежуточных результатов в механизме внимания (attention)

LLM для программирования часто содержат несколько "экспертов", специализирующихся на разных языках программирования. Эта особенность архитектуры говорит о том, что чаще всего LLM обучают не программированию вообще, а скорее генерации кода на конкретных языках программирования. В зависимости от dataset-а, LLM может лучше генерировать код на одном языке программирования, но существенно хуже на другом. Также в этом факте есть потенциальное ограничение для развития индустрии программирования - для популярных языков программирования dataset больше, генерация кода лучше и это создаёт высокий порог вхождения для новых языков. Если LLM для кодирования отлично генерирует код JavaScript, но отвратительно на Lua, то мотивация программистов выбирать Lua для своих новых проектов значительно уменьшается. Со временем может создаться ситуация, когда будут зафиксированы некоторые доминирующие языки программирования, которые будут блокировать появление инноваций в области описания приложений.

### Как размер контекста замедляет работу

Длина контекста — это максимальное количество токенов (слов или подслов), которые модель может обработать за один запрос.

Обработка каждого дополнительного токена требует вычислений для:

- токенизации (разбиения текста на токены)
- расчёта весов внимания между всеми токенами в контексте
- кодирования отношений между токенами
- генерации ответа

Сложность вычислений в механизмах внимания (attention) часто растёт квадратично относительно длины контекста: если контекст увеличивается в n раз, время обработки может вырасти в n ^ 2 раз (квадрат).

Больше токенов = больше времени на:

- передачу данных (в т. ч. между CPU и GPU)
- выполнение вычислений на каждом слое нейронной сети
- генерацию последовательности выходных токенов

Длинный контекст требует:

- больше видеопамяти (VRAM) для хранения активаций и матриц внимания
- большего объёма оперативной памяти (RAM) для промежуточных вычислений

Это может привести к:

- замедлению из‑за подкачки данных
- необходимости использовать более дорогое оборудование

Таким образом, увеличение размера контекста замедляет работу LLM из‑за:

- **квадратичного роста** вычислительных затрат
- повышенной нагрузки на память
- дополнительных механизмов обработки длинных последовательностей

>Новость на 3DNews [Санкции не помогли: ИИ-модель китайской Z.ai, обученная на чипах Huawei, заняла лидирующие позиции в рейтингах](https://3dnews.ru/1143921/sanktsii-ne-pomogli-iimodel-kitayskoy-zai-obuchennaya-na-chipahhuawei-zanyala-lidiruyushchie-pozitsii-v-reytingah):
>
>GLM-5.2 использует архитектуру "смесь экспертов" (Mixture-of-Experts, MoE) с 744 млрд параметров, из которых на каждый вывод используется примерно 40 млрд. Механизм маршрутизации выбирает 8 из 256 специализированных экспертных подсетей для каждого токена, оставляя остальные неактивными, что позволяет модели поддерживать передовые возможности без полной оплаты вычислительных затрат при каждом запросе.
>
>Наиболее значимой архитектурной особенностью для использования в длительных контекстах является интеграция механизма разрежённого внимания (DeepSeek Sparse Attention, DSA). Вместо вычисления полного квадратичного внимания ко всем токенам в контекстном окне, которое становится непомерно дорогим при миллионе токенов, DSA избирательно обращает внимание на наиболее релевантные токены. Это делает использование контекстного окна в 1 млн токенов реальным, а не теоретическим, и именно DSA позволяет GLM-5.2 обрабатывать весь большой код за один проход вывода

Выбор размера контекста должен учитывать задачу. Для быстрых ответов хватит 4–16k токенов, для глубокого анализа — 128k+, но с готовностью к задержкам.

При очень длинных контекстах могут включаться дополнительные механизмы:

- суммирование части контекста
- использование RAG (Retrieval‑Augmented Generation) для извлечения релевантной информации
- разбиение на чанки (chunking) и их последовательная обработка

Каждый такой шаг добавляет задержки.

| Размер контекста | Типичное время ответа | Сценарии использования                         |
|------------------|-----------------------|------------------------------------------------|
| 4k токенов       | Быстро (200–500 мс)   | Простые чат‑боты, FAQ, короткие запросы        |
| 128k токенов     | Умеренно (1–5 с)      | Анализ документов, код, длинные диалоги        |
| 1M токенов       | Медленно (5–30 с+)    | Глубокий анализ данных, мультимодальные задачи |

Пример: модель с окном в 4k токенов обработает 10‑страничный документ за один проход, а модель с 128k — сможет проанализировать 100‑страничный отчёт целиком. Но второй запрос будет выполняться в разы дольше.

```shell
ollama launch opencode
```

Причина: консольная opencode и OpenCode Desktop - разные приложения.

Установка консольного агента OpenCode может быть осуществлена через npm. Для этого сначала необходимо обновить npm. Команду нужно выполнять от учётной записи администратора, т.к. пакет устанавливается глобально:

```shell
npm install -g npm@11.16.0
```

Команда установки opencode через npm не была успешной:

```shell
npm i -g opencode-ai
```

Ошибка:

```shell
npm error code MODULE_NOT_FOUND
npm error Cannot find module 'negotiator'
npm error Require stack:
```

Установка выполнилась успешно после обновления Node.js до версии 24.

После установки консольной версии OpenCode удалось запустить приложение командой: `ollama launch opencode`. При запуске приложение требует выбрать LLM. По данным контрольной панели - работает локальная Gemma 4.

Для полноценной работы с приложением имеет смысл открыть его на полный экран. В этом случае, в правой части экрана появляется полезная информация о приложении, в том числе, размер контекста.

Для анализа файлов на диске необходимо явным образом указать папки и предоставить права доступа к ним.

Система работает и даёт разумные предложения по модификации кода. Однако размер контекста очень быстро доходит до десятков тысяч токенов. Если не настроить размер данных, то часть контекста запроса может быть утеряна.

Получить список установленные глобально пакетов можно командой:

```shell
npm list -g --depth=0
```

Проверить устаревшие пакеты можно командой:

```shell
npm outdated -g
```

Удалить пакет можно командой:

```shell
npm uninstall -g opencode-ai
```

## Пример структуры промпта

Gemma 4 сгенерировала мне следующий пример промпта для запроса на улучшение качества кода:

```prompt
[SETUP: Acknowledge the files here]
I have provided several files (the attached project) that handle [DESCRIBE FUNCTIONALITY]. The code currently works, but I suspect it is inefficient and lacks modern best practices.

[GOAL: State the area of improvement]
My primary goals are: 
1. Improve performance/speed.
2. Increase readability and maintainability.
3. Add specific feature: [IF THERE IS A NEW FEATURE, LIST IT].

[TASK: Define the analysis steps]
Please act as a Senior Software Architect. Review all files and provide the following:
1. **Key Areas for Improvement:** Identify 3-5 specific places in the code that are inefficient (e.g., redundant database queries, nested loops, poor variable naming).
2. **Refactored Code:** Provide a fully refactored version of the files. Do not just fix the problems; show me how to make the *whole system* cleaner.
3. **Explanation:** For each major change you made, explain *why* it is an improvement (e.g., "I replaced the try/catch block with a dedicated result object to improve type safety.").

[CONSTRAINTS: Guide the AI's output]
Use separate code blocks for the suggested refactoring. Keep the explanation concise but highly technical.
```
